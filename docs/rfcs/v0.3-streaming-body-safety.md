# RFC: Lafetch v0.3 Streaming과 본문 안전성

상태: 제안

대상 버전: `0.3.0-alpha`

## 요약

Lafetch의 기존 소비 경로는 하나의 Builder를 여러 소비자가 안전하게 사용할 수 있도록 최종 응답을 메모리에 보관합니다. 이 계약은 일반 JSON과 짧은 응답에는 편리하지만, 크기를 알 수 없는 응답에서는 메모리 사용량이 제한되지 않습니다.

v0.3은 다음 두 실행 방식을 명시적으로 분리합니다.

- 기존 `await`, `response()`, `raw()`는 **크기 제한이 있는 Buffered 실행**을 사용합니다.
- 새로운 `stream()`은 **본문을 보관하지 않는 단일 소비 Streaming 실행**을 사용합니다.

두 방식 사이에서 반환 타입을 자동 전환하지 않습니다. 응답이 크다는 이유로 데이터 반환값이 갑자기 Stream으로 바뀌는 동작은 허용하지 않습니다.

## 목표

- 일반 요청의 데이터 우선 문법을 변경하지 않습니다.
- Buffered 응답에 기본 크기 상한을 적용합니다.
- 실제 읽은 바이트를 추적하여 없거나 잘못된 `Content-Length`에 의존하지 않습니다.
- Streaming 응답을 Fetch 표준 `Response`와 `ReadableStream`으로 제공합니다.
- Timeout과 Abort가 응답 헤더뿐 아니라 본문 종료까지 적용되도록 합니다.
- 본문 일부를 전달한 뒤 Retry하여 데이터가 중복되는 일을 방지합니다.
- Cache, Deduplication, Feature finalizer의 Streaming 규칙을 고정합니다.
- Browser, Node.js, Next.js, Workers/Edge에서 같은 계약을 검증합니다.

## 비목표

- Node.js 전용 Stream API를 코어에 추가하지 않습니다.
- JSON token Streaming이나 NDJSON 파서를 코어에 포함하지 않습니다.
- 다운로드 파일 저장, 진행률 UI, SSE 파서는 별도 Helper나 사용자 코드의 책임입니다.
- Cache와 Deduplication의 Streaming multicast를 구현하지 않습니다.
- Streaming 응답을 자동으로 Buffering하지 않습니다.
- 업로드 진행률 API는 이번 범위에 포함하지 않습니다.

## 공개 API 제안

### Streaming 종결 메서드

```ts
interface RequestBuilder<TData> {
  stream(): Promise<Response>;
}
```

`stream()`은 상태와 헤더가 확정되면 표준 Fetch `Response`를 반환합니다. `response.body`는 네트워크 본문과 연결된 단일 소비 `ReadableStream<Uint8Array> | null`입니다.

```ts
const response = await api
  .get("/downloads/report.csv")
  .timeout("2m")
  .stream();

const reader = response.body?.getReader();

while (reader) {
  const { done, value } = await reader.read();
  if (done) break;
  consume(value);
}
```

Text Stream은 Web Platform 변환기를 그대로 사용합니다.

```ts
const response = await api
  .get("/events")
  .stream();

const text = response.body?.pipeThrough(new TextDecoderStream());
```

Lafetch 전용 Stream wrapper를 만들지 않습니다. 브라우저와 Edge의 `ReadableStream`, Node.js의 Web Stream 호환 기능을 그대로 사용할 수 있습니다.

### Buffered 본문 크기

```ts
export type ByteSize =
  | number
  | `${number}b`
  | `${number}kb`
  | `${number}mb`
  | `${number}gb`;

interface RequestBuilder<TData> {
  maxResponseSize(limit: ByteSize): RequestBuilder<TData>;
}
```

숫자는 Byte, 문자열 단위는 1024 배수로 해석합니다. 값은 유한한 0 이상의 안전한 정수 Byte로 정규화되어야 합니다.

```ts
const report = await api
  .get<Report>("/reports/1")
  .maxResponseSize("2mb");
```

Buffered 실행의 기본 상한은 `10mb`입니다. 기본값을 제거하거나 무제한으로 설정하는 API는 제공하지 않습니다. 큰 응답은 유한한 상한을 명시하거나 `stream()`으로 소비합니다.

```ts
const response = await api
  .get("/backups/latest")
  .maxResponseSize("1gb")
  .stream();
```

Streaming은 기본적으로 전체 크기를 제한하지 않지만, `maxResponseSize()`를 명시하면 전달된 실제 바이트를 세어 같은 상한을 적용합니다.

## 소비 방식 비교

| 종결 방식 | 본문 보관 | 다중 소비 | 반환값 | 기본 크기 상한 |
| --- | --- | --- | --- | --- |
| 직접 `await` | Buffered | 가능 | 디코딩된 데이터 | `10mb` |
| `response()` | Buffered | 가능 | `LafetchResponse<T>` | `10mb` |
| `raw()` | Buffered | 가능 | 복제 가능한 `Response` | `10mb` |
| `stream()` | Streaming | 불가 | 본문이 연결된 `Response` | 없음 |

`raw()`는 Fetch `Response`를 반환하지만 기존과 같이 보관된 본문의 복제본입니다. 진짜 Streaming이 필요하면 반드시 `stream()`을 사용합니다.

## Builder 소비 규칙

하나의 Builder는 Buffered 또는 Streaming 중 하나의 실행 방식만 가질 수 있습니다.

- Buffered 소비자는 기존처럼 같은 실행 결과의 독립적인 복제본을 사용합니다.
- 첫 `stream()` 호출이 Builder의 Streaming 실행을 점유합니다.
- 같은 Builder에서 `stream()`을 다시 호출하면 `HttpConsumptionError`가 발생합니다.
- 첫 호출이 반환한 Promise 자체를 공유할 수는 있지만, 완료되는 `Response.body`는 Web Stream 규칙에 따라 한 번만 소비할 수 있습니다.
- Streaming을 시작한 Builder를 직접 `await`하거나 `response()`, `raw()`, `then()`, `catch()`, `finally()`로 다시 소비하면 `HttpConsumptionError`가 발생합니다.
- Buffered 소비를 먼저 시작한 Builder에서 `stream()`을 호출해도 `HttpConsumptionError`가 발생합니다.
- 혼합 소비는 두 번째 Transport 실행으로 우회하지 않습니다.

다음 코드는 허용하지 않습니다.

```ts
const request = api.get("/large-file");

const streaming = request.stream();
const buffered = request.raw(); // HttpConsumptionError
```

Builder를 변형하면 별도의 실행 식별자를 가지므로, 정말 두 요청이 필요하면 명시적으로 새 Builder를 만듭니다.

## `as()`, `validate()`, `mapError()`

`as()`와 `validate()`는 Buffered 소비 전용입니다. 두 설정 중 하나가 명시된 Builder에서 `stream()`을 호출하면 Transport 실행 전에 `HttpConfigurationError`가 발생합니다. Streaming에서 자동 디코딩이나 전체 본문 스키마 검증을 암묵적으로 수행하지 않습니다.

```ts
api.get("/events").as("text").stream(); // 허용하지 않음
api.get("/events").validate(schema).stream(); // 허용하지 않음
```

`mapError()`는 Streaming에서도 유지합니다.

- Response가 반환되기 전 실패는 기존 요청 오류 흐름에서 처리합니다.
- 본문 읽기 실패는 `phase: "response"`로 Mapper를 거친 뒤 `reader.read()`를 거부합니다.
- 본문이 이미 전달된 후에는 `stream()` Promise를 다시 거부할 수 없습니다.

Streaming 오류 Mapper의 `response`에는 상태와 헤더를 보존한 body 없는 Response snapshot을 전달합니다.

## 상태 코드와 Retry

Retry는 사용자에게 accepted response body를 노출하기 전까지만 가능합니다.

1. Transport 또는 Interceptor가 Response를 반환합니다.
2. `afterResponse` Feature와 상태 코드 판정을 실행합니다.
3. 재시도 대상 상태라면 본문을 취소하고 다음 시도를 시작합니다.
4. 최종 상태가 허용되면 Streaming Response를 사용자에게 노출합니다.
5. 이후 발생한 본문 오류나 Attempt Timeout은 Stream을 종료하지만 Retry하지 않습니다.

본문 일부를 사용자에게 전달한 뒤 새 응답으로 교체하면 중복 데이터와 파일 손상이 발생할 수 있으므로 금지합니다.

최종 상태가 허용되지 않으면 `stream()`은 `HttpStatusError`를 발생시키고 본문을 취소합니다. 이때 오류의 `response`는 상태와 헤더를 보존하지만 body는 포함하지 않습니다. 오류 본문을 읽어야 하는 사용자는 해당 상태를 명시적으로 허용합니다.

```ts
const response = await api
  .get("/exports/1")
  .acceptStatus((status) => status === 200 || status === 422)
  .stream();
```

Buffered 실행의 `HttpStatusError.response`는 크기 상한 안에서 기존처럼 독립적으로 소비할 수 있습니다.

## Timeout과 Abort

전체 Timeout, Attempt Timeout, 사용자 `AbortSignal`은 본문이 완료될 때까지 살아 있습니다.

| 상황 | Streaming 결과 |
| --- | --- |
| 전체 Timeout | 본문 읽기가 `HttpTimeoutError(scope: "total")`로 실패 |
| Attempt Timeout | 본문 읽기가 `HttpTimeoutError(scope: "attempt")`로 실패, Retry 없음 |
| 사용자 Abort | 본문 읽기가 `HttpAbortError`로 실패 |
| 소비자의 `body.cancel(reason)` | 원본 본문 취소, lifecycle finalization 실행 |

Timeout이나 Abort가 발생하면 원본 reader를 취소하고 관련 signal listener와 timer를 정리합니다.

Streaming Response를 끝까지 읽지 않는 소비자는 `response.body?.cancel()`을 호출해야 합니다. Timeout이 없는 요청을 읽지도 취소하지도 않으면 네트워크와 Feature 수명이 열린 상태로 남을 수 있습니다.

## 본문 크기 상한

크기 상한은 Fetch가 사용자에게 제공하는 실제 응답 body Byte를 기준으로 합니다.

- 유효한 `Content-Length`가 상한보다 크면 읽기 전에 실패할 수 있습니다.
- `Content-Length`가 없거나 실제보다 작아도 각 chunk의 Byte를 누적합니다.
- 압축 응답은 Fetch가 해제한 뒤 전달하는 실제 body 크기를 기준으로 제한합니다.
- 제한을 넘기는 chunk는 Lafetch의 보관 Buffer에 추가하지 않습니다.
- 초과 시 원본 body를 취소합니다.

상한은 전체 JavaScript heap이 아니라 **Lafetch가 보관하거나 전달한 응답 body payload Byte**에 대한 계약입니다. Fetch 구현이 이미 할당한 한 개의 chunk, Response와 Header 객체, 디코더 및 사용자 코드의 메모리는 이 값에 포함되지 않습니다.

## 오류 모델

새 오류를 추가합니다.

```ts
class HttpResponseTooLargeError extends HttpConsumptionError {
  readonly code: "ERR_HTTP_RESPONSE_TOO_LARGE";
  readonly limitBytes: number;
  readonly receivedBytes: number;
  readonly declaredBytes?: number;
}
```

`receivedBytes`는 실제로 읽은 Byte, `declaredBytes`는 유효한 `Content-Length`가 제공된 경우의 선언값입니다. Header만으로 조기 거부하면 `receivedBytes`는 `0`일 수 있습니다.

- Buffered 실행에서는 크기 초과 시 데이터, `response()`, `raw()`가 이 오류로 실패합니다.
- Streaming에서 명시적 상한을 사용하면 한계를 넘는 `reader.read()`가 이 오류로 실패합니다.
- 이 오류는 Retry 대상이 아닙니다.
- `mapError()`에는 `phase: "response"`로 전달합니다.

잘못된 `ByteSize`나 0 미만, 무한대, 안전한 정수 범위를 넘는 값은 선언 시 `HttpConfigurationError`를 발생시킵니다.

## Cache와 Deduplication

`cache()`와 `dedupe()`는 `stream()`과 함께 사용할 수 없습니다.

```ts
api.get("/video").cache("1m").stream(); // HttpFeatureConflictError
api.get("/video").dedupe().stream(); // HttpFeatureConflictError
```

이유는 다음과 같습니다.

- Cache 저장은 완료된 전체 본문을 소유해야 합니다.
- Streaming Cache는 사용자 소비와 저장소 쓰기 사이의 tee 및 backpressure 정책이 필요합니다.
- Deduplication follower에게 같은 Stream을 전달하면 소비자 속도 차이만큼 메모리가 쌓일 수 있습니다.
- 한 소비자의 취소가 다른 소비자에게 미치는 규칙도 별도 multicast 계약이 필요합니다.

Buffered Cache와 Deduplication에는 같은 기본 `10mb` 상한이 적용됩니다. 공식 Cache는 상한 검증이 끝난 응답만 저장하고 Deduplication follower도 제한된 retained response만 받습니다.

## Feature 생명주기

Streaming에서도 `prepare`, `beforeAttempt`, `intercept`, `afterResponse`, `onAttemptError`, `mapError`, `onEvent`, `finalize` 순서는 유지합니다.

차이는 최종 완료 시점입니다.

- `stream()` Promise는 accepted response의 상태와 헤더가 준비되면 완료됩니다.
- `request:success`와 `finalize`는 body가 끝까지 닫힌 뒤 실행됩니다.
- body 읽기 오류, Timeout, Abort, consumer cancel은 실패 또는 취소 경로로 finalization을 실행합니다.
- finalizer가 실패하면 마지막 `reader.read()` 또는 `body.cancel()`이 `HttpFeatureError`로 실패할 수 있습니다.
- body가 없는 `HEAD`, `204`, `205` Response는 `stream()`이 반환되기 전에 finalization을 완료합니다.

공식 Telemetry는 body를 읽지 않으며 기존처럼 요청 결과와 격리됩니다. `attempt:response`는 헤더 수신 시점, 최종 `request:success` 또는 `request:error`는 body 종료 시점에 발생합니다.

사용자 Feature가 Streaming Response를 clone하거나 body를 읽으면 Web Stream tee의 buffering 책임은 해당 Feature 작성자에게 있습니다. v0.5 Feature SDK에서 이를 검사할 conformance 규칙을 추가합니다.

## Streaming 업로드

새로운 업로드 API는 추가하지 않습니다.

```ts
await api
  .post("/upload")
  .body(stream);
```

기존 `ReadableStream` body는 한 번만 보낼 수 있습니다. Retry가 필요하면 시도마다 새 Stream을 반환하는 `bodyFactory()`를 사용합니다.

```ts
await api
  .post("/upload")
  .bodyFactory(() => createUploadStream())
  .retry(1, { methods: ["POST"] });
```

요청 body는 Lafetch가 전체 Buffer로 변환하지 않습니다.

## 구현 순서

1. `ByteSize` 정규화와 `HttpResponseTooLargeError`를 추가합니다.
2. 기존 clone-and-drain 보관 경로를 실제 Byte를 세는 bounded retention으로 교체합니다.
3. Buffered 응답 snapshot에서 소비자별 독립 Response를 생성합니다.
4. Builder에 Buffered/Streaming 소비 방식 claim을 추가합니다.
5. Streaming executor와 body lifecycle wrapper를 구현합니다.
6. Timeout, Abort, body 오류와 `mapError()`를 연결합니다.
7. Cache와 Deduplication 충돌 및 공식 Cache write 순서를 고정합니다.
8. Feature finalization과 최종 이벤트를 body 종료 시점으로 이동합니다.
9. 런타임 및 패키지 계약 테스트를 추가합니다.

## 테스트 계약

### Buffered

- 기본 `10mb` 이하 JSON, text, binary, FormData
- `Content-Length`가 상한보다 큰 응답
- Header가 없거나 실제보다 작은 응답
- chunk 경계에서 정확히 상한과 상한 + 1 Byte
- `raw()`와 여러 소비자의 독립성
- Cache 저장 전 상한 적용
- Deduplication leader와 follower의 독립 소비

### Streaming

- 첫 Byte가 전체 body 완료 전에 전달되는지 검증
- 전체 body가 메모리에 보관되지 않는지 검증
- 정상 완료, reader cancel, source error
- 전체 Timeout, Attempt Timeout, 사용자 Abort
- body 노출 이후 Retry가 발생하지 않는지 검증
- `as()`, `validate()`, Cache, Deduplication 충돌
- 반복 `stream()` 호출 거부와 Buffered 혼합 소비
- finalizer 및 lifecycle event 시점
- 명시적 `maxResponseSize()` 초과

### 런타임

- Chromium의 `ReadableStream`과 `TextDecoderStream`
- Node.js 20, 22, 24의 Web Stream
- Next.js App Router Route Handler와 Server Component
- `workerd` 격리 환경
- 실제 npm tarball의 타입 및 JavaScript 소비

## 거부한 대안

### `.as("stream")`

`as()`는 Buffered 응답의 디코더를 선택합니다. Stream은 디코더가 아니라 실행과 소비 생명주기 자체를 바꾸므로 같은 selector에 넣지 않습니다.

### `raw()`를 Streaming으로 변경

기존 `raw()`는 memoized 실행의 독립적인 Response clone을 제공하며 여러 번 호출할 수 있습니다. 이를 live body로 바꾸면 v0.2 소비 계약이 조용히 깨집니다.

### 크기에 따라 자동으로 Stream 반환

동일한 코드의 반환 타입과 오류 시점이 응답 크기에 따라 달라지므로 허용하지 않습니다.

### Streaming Cache와 Deduplication

multicast backpressure, 느린 follower, 부분 Cache entry, 취소 전파를 별도 설계하지 않고 Response tee에 맡기면 메모리 상한을 보장할 수 없습니다.

### Buffered 무제한 옵션

실수로 무제한 응답을 보관하는 경로를 다시 만들기 때문에 제공하지 않습니다. 큰 응답은 명시적인 유한 상한 또는 `stream()`을 사용합니다.

## 완료 조건

- 기존 데이터 우선 API와 Buffered 다중 소비 테스트가 유지되어야 합니다.
- 모든 Buffered 응답은 기본 또는 요청별 Byte 상한을 적용받아야 합니다.
- Streaming에서 Lafetch가 전체 body를 retained response로 보관하지 않아야 합니다.
- Timeout과 Abort가 body 종료까지 유지되고 모든 timer와 listener가 정리되어야 합니다.
- body 노출 이후 Retry가 발생하지 않아야 합니다.
- Cache와 Deduplication이 Streaming과 암묵적으로 결합되지 않아야 합니다.
- 오류 코드와 `mapError()` phase가 TypeScript와 JavaScript 계약 테스트로 고정되어야 합니다.
- Node.js, Chromium, Next.js, Workers/Edge CI가 통과해야 합니다.

## 확정이 필요한 항목

이 RFC를 승인하기 전에 다음 네 가지를 최종 확인합니다.

1. Streaming 종결 메서드를 `stream()`으로 확정할지
2. Buffered 기본 상한을 `10mb`로 확정할지
3. `maxResponseSize()`를 Buffered와 Streaming 양쪽에 적용할지
4. 반복 `stream()` 호출을 `HttpConsumptionError`로 거부할지
