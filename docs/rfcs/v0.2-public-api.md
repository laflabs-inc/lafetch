# RFC: Lafetch v0.2 공개 API

상태: 승인, v0.2.0-alpha 안정화 중

## 제품 정의

Lafetch는 요청 프레임워크가 아니라 TypeScript HTTP 클라이언트다.

> 데이터는 바로 받고, 실패 정책은 읽기 쉽게 선언한다.

Feature Runtime, Capability 해석, Transport 실행 계획은 이 목표를 구현하는 내부 구조다. 일반 사용자가 Lafetch를 이해하기 위해 먼저 학습해야 하는 제품 개념이 아니다.

## 목표

- 가장 흔한 JSON 요청은 Axios보다 짧고 결과가 직접적이어야 한다.
- 요청 설정과 안정성 정책은 한 방향으로 읽혀야 한다.
- 하나의 동작에는 하나의 공식 표현만 제공한다.
- 전체 응답, 원본 Response, 사용자 정의 Feature는 명시적으로 선택한다.
- 안전한 재시도, 상태 격리, 오류 구분은 간단한 API 뒤에서도 유지한다.

## 공식 문법

```text
lafetch.create() → api.method(url) → body/config/policy → await
```

```ts
const api = lafetch.create({
  baseUrl: "https://api.example.com",
});

const user = await api.get<User>("/users/123");
```

`lafetch`는 `create()`만 노출한다. 모든 요청은 명시적인 클라이언트에 속한다.

## 데이터 우선 소비

Builder를 직접 `await`하면 자동 디코딩된 데이터 `T`를 반환한다.

```ts
const user = await api.get<User>("/users/123");
```

상태 코드, 헤더, 요청 메타데이터가 필요할 때만 `response()`를 사용한다.

```ts
const response = await api
  .get<User>("/users/123")
  .response();

response.data;
response.status;
response.headers;
response.meta.attempts;
```

Fetch `Response`가 직접 필요하면 `raw()`를 사용한다. `raw()`는 데이터 디코딩과 스키마 검증을 적용하지 않는다.

`send()`과 응답 종결 메서드 `json()`, `text()`, `blob()`, `arrayBuffer()`, `formData()`는 제거한다. 응답 형식 강제는 실행 전 `as()`로 선언한다.

```ts
const text = await api.get<string>("/health").as("text");
const file = await api.get<Blob>("/files/1").as("blob");
```

기본값 `as("auto")`는 `Content-Type`을 기준으로 JSON, 문자열, `ArrayBuffer`를 선택한다.

`as()`는 `"auto"`, `"json"`, `"text"`, `"arrayBuffer"`, `"blob"`, `"formData"`만 허용한다. TypeScript뿐 아니라 JavaScript 런타임에서도 닫힌 값을 검증하며, 잘못된 값은 Transport 실행 전에 `HttpConfigurationError`를 발생시킨다.

## 요청 본문

JSON 요청 본문은 `json(value)`로 설정한다.

```ts
const user = await api
  .post<User>("/users")
  .json({ name: "Dohyun" });
```

원시 `BodyInit`은 `body(value)`, 재시도마다 새 본문이 필요한 경우에만 `bodyFactory(factory)`를 사용한다. `jsonBody()`는 제거한다.

## 응답 검증과 오류 매핑

응답 스키마는 동작을 설명하는 `validate(schema)`로 통일한다.

```ts
const user = await api
  .get("/users/123")
  .validate(userSchema);
```

`mapError()`는 Transport, HTTP 상태, 디코딩, 스키마 검증 실패에 동일하게 적용한다. 재시도 판단은 오류 매핑 전에 끝나므로 도메인 오류 변환이 재시도 안전성을 바꾸지 않는다. `mapDecodeError()`는 제거한다.

## Timeout과 Retry

전체 제한 시간과 개별 시도 제한 시간을 서로 다른 이름으로 표현한다.

```ts
await api
  .get<User>("/users/123")
  .timeout("10s")
  .attemptTimeout("3s");
```

`timeout({ total, attempt })` 형태는 제거한다.

Retry의 첫 번째 인자는 실패 후 추가로 재시도할 횟수다.

```ts
await api.get<User>("/users/123").retry(2); // 최초 1회 + 재시도 2회
```

고급 설정도 항상 같은 시그니처를 사용한다.

```ts
await api.get<User>("/users/123").retry(2, {
  backoff: {
    type: "exponential",
    base: "200ms",
    max: "10s",
    jitter: "full",
  },
});
```

Backoff `type`은 `"fixed"`와 `"exponential"`, `jitter`는 `"none"`과 `"full"`만 허용한다. 문자열 축약형이나 알 수 없는 값은 기본 정책으로 대체하지 않는다.

숫자와 `{ attempts }` 객체로 같은 동작을 표현하던 기존 API는 제거한다.

Cache도 TTL을 첫 번째 인자에 명시하고 고급 옵션은 두 번째 인자에만 둔다. 기본 TTL을 숨기거나 옵션 객체로 같은 동작을 표현하지 않는다.

```ts
await api.get<User[]>("/users").cache("30s");
await api.get<User[]>("/users").cache("30s", { store });
```

Telemetry는 Handler를 항상 첫 번째 인자로 받는다. 여러 수집기를 구분하는 이름만 두 번째 옵션에 둔다. 공식 Telemetry는 관찰 전용이며 Handler 실패가 HTTP 요청의 결과를 바꾸지 않는다.

```ts
await api.get<Health>("/health").telemetry(onEvent, {
  name: "metrics",
});
```

## 클라이언트 경계

`create()` 옵션은 공유 환경에만 사용한다.

- `baseUrl`
- `headers`
- `credentials`
- `transport`
- 특수 런타임을 위한 `runtime`

Credentials는 Fetch 표준의 `"omit"`, `"same-origin"`, `"include"`만 허용하며 클라이언트와 요청 양쪽에서 런타임 검증한다.

클라이언트 단위 `features`와 `extend()`는 제거한다. Cache와 진행 중 Deduplication의 소유자는 항상 `create()`로 생성된 클라이언트다.

사용자 정의 HTTP 메서드는 옵션 객체 대신 명시적인 시그니처를 사용한다.

```ts
await api.request<Result>("PURGE", "/cache/entries");
```

Named HTTP 메서드와 마찬가지로 나머지 설정은 Builder에서 이어간다.

## Feature 경계

공식 기능은 `.timeout()`, `.retry()`, `.cache(ttl)`처럼 전용 메서드로 제공한다. 사용자 정의 기능만 `.use(feature)`를 사용한다.

Feature Runtime 타입과 Helper는 루트가 아닌 고급 진입점에서 제공한다.

```ts
import { defineFeature } from "@laflabs/lafetch/feature";
```

한 요청에서 Feature 이름은 고유해야 한다. 같은 공식 정책을 두 번 구성하거나 사용자 Feature로 기존 이름을 덮어쓰면 마지막 설정을 암묵적으로 선택하지 않고 네트워크 실행 전에 충돌 오류를 발생시킨다.

루트 패키지는 일반 요청에 필요한 클라이언트, 응답, Transport, 오류 타입만 노출한다.

## 마이그레이션

전체 전환 순서와 동작 차이는 [v0.1에서 v0.2로 마이그레이션](../migration-v0.2.md)에 정리한다.

| v0.1 | v0.2 |
| --- | --- |
| `await request` → `HttpResult<T>` | `await request` → `T` |
| `request.send()` | `request.response()` |
| `request.jsonBody(value)` | `request.json(value)` |
| `request.json<T>()` | `await request` 또는 `.as("json")` |
| `request.text()` | `request.as("text")` |
| `request.schema(schema)` | `request.validate(schema)` |
| `request.mapDecodeError()` | `request.mapError()` |
| `request.timeout({ total, attempt })` | `.timeout(total).attemptTimeout(attempt)` |
| `request.retry({ attempts: 3 })` | `request.retry(2)` |
| `request.cache()` 또는 `.cache({ ttl })` | `request.cache(ttl, options?)` |
| `request.telemetry({ onEvent })` | `request.telemetry(onEvent, options?)` |
| `api.request(url, { method })` | `api.request(method, url)` |
| `api.extend(options)` | `lafetch.create(options)` |
| 클라이언트 `features` | 요청별 전용 메서드 또는 `.use(feature)` |
| 루트의 Feature 타입 | `@laflabs/lafetch/feature` |

## 공개 전 안정화 조건

v0.2 API 구현 후 다음 조건을 통과해야 첫 공개 버전 후보가 된다.

1. 명시적인 Streaming 실행 경로와 메모리 상한
2. JSON, text, binary, FormData, 빈 응답 소비 계약
3. Abort, 전체 Timeout, 시도 Timeout, Retry 경쟁 상태 검증
4. Cache와 Deduplication의 클라이언트·테넌트 격리 검증
5. 실제 패키지 설치와 export condition 소비자 테스트
6. Browser, Node.js, Next.js, Workers/Edge 런타임 매트릭스
7. 루트 export와 번들 크기 회귀 예산
